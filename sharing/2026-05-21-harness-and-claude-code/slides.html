// 2. definition

什么是 Harness？

Harness = 包裹在 LLM 外面的程序化框架 · 管理 Agent 的运行环境、上下文、状态、工具和生命周期。

"Humans steer. Agents execute."  — Ryan Lopopolo, OpenAI · 2026-02

定义先讲清楚。Harness——脚手架——是包裹在 LLM 外面的程序化框架。它不改变模型本身，但通过工程手段让模型在复杂任务中接近能力上限。

类比讲：把 LLM 想象成一个能力出众的工程师，他写代码很快、思考很周全。但同一个工程师单干，跟在一个完善的团队里干，产出质量天差地别。

团队里有什么？项目管理、代码仓库、CI/CD、Code Review、协作协议。这些不是写代码的能力，但它们决定了写代码的人能发挥到什么水平。

Harness 就是 LLM 的"团队"。

OpenAI 的 Ryan Lopopolo 给了一句最精炼的总结——"Humans steer. Agents execute." 人定方向，Agent 做事。这是 2026 年新范式的核心。

// 2. diagnosis

为什么"写更好的 prompt"不够？

Anthropic 和 OpenAI 各自从产品实践中诊断出三大病根：

①
Context Anxiety · 上下文焦虑

模型接近上下文窗口上限时会"begin wrapping up work prematurely"——提前收尾、草草了事。

②
Self-Evaluation Bias · 自评偏差

"Agents tend to respond by confidently praising the work — even when, to a human observer, the quality is obviously mediocre."

③
Session Amnesia · 会话失忆

"Each new session begins with no memory of what came before." 每次开新会话都失忆。

"Early progress was slower than we expected, not because Codex was incapable, but because the environment was underspecified. The fix was almost never 'try harder.'" — Ryan Lopopolo, OpenAI · 2026-02

为什么"prompt 写得再好"也解决不了问题？两家公司的诊断刚好互补。

Anthropic 的 Justin Young 和 Prithvi Rajasekaran 提了三个内在缺陷——

第一是 Context Anxiety，上下文焦虑——模型快没上下文了就开始草草收尾。

第二是 Self-Evaluation Bias，自评偏差——这个最致命。Agent 评自己写的烂代码会自信地说"完美"。

第三是 会话失忆——每次开新会话都从零开始，背景要重新讲。

这三个都是模型本身的限制，不是 prompt 能解决的。

OpenAI 的 Ryan Lopopolo 一句话总结得更狠——"Codex 不是不够聪明，是环境定义不充分。" 当 Agent 失败时，正确的反应不是"再试一次"，而是问"它缺哪个能力？这个能力怎么让它看见？"

这三大病根加这一句诊断，自然引出 harness 这个结构性方案。

这页给大家看 harness 的解剖图。Claude Code 的实际实现有 5 层，从底到上——

L1 基础设施：Bootstrap、Context、Skills、Memory。这一层管"Agent 启动时知道什么、看到什么、记得什么"。

L2 控制层：Hook 系统。在 prompt 提交、工具调用前后注入逻辑——比如自动跑 linter、自动加 git 提交。

L3 协作层：Coordinator。主 Agent 派活，sub-Agent 干活，每个 sub-Agent 只能看自己被授权的工具子集。这一层是 Anthropic V2 那个 Generator-Evaluator 架构落到工程里的样子。

L4 任务层：Task System，6 种任务类型——Shell（执行命令）、Agent（子 Agent）、Teammate（异步同事）、Remote（远程任务）、Dream（后台规划）、Workflow（多步骤）。

L5 协议层：App Server 加 MCP 协议。这一层让一个 harness 能被多个客户端共享——你在 VS Code、Web、CLI 里看到的是同一个 Claude。

这页不展开。今天 demo 重点用 L1 的 Memory 和 Skills，加 L3 的 Coordinator——这两层是研究员最容易上手、收益最大的入口。

之前讲的都是工业博客（Anthropic / OpenAI），有人会问"是不是只是大厂自吹"。这一页给学术层面的反驳。

2026 年 TMLR 上有一篇 survey 论文叫 Agent Harness Engineering: A Survey，作者来自 CMU、Yale、JHU、Amazon 等机构。它提出一个核心论断——"binding-constraint thesis"，意思是：对长程 agent 任务，benchmark 表现的差异主要来自 harness，不是来自模型。

论文给了 3 个最近半年的硬数据来支持——

第一：Bölük 2026 只改了 edit-tool 的格式（怎么让 Agent 编辑文件），跨 15 个模型测，得到了10 倍的编程 benchmark 提升。模型权重没动。

第二：LangChain 的 DeepAgents 团队，固定 GPT-5.2-Codex 不变，只通过改 system prompt、加 middleware 注入上下文、加自验证 hook，把 Terminal-Bench 2.0 从 52.8% 拉到 66.5%——13.7 个百分点。

第三：Stanford 和 MIT 的 Meta-Harness 工作，做自动化 harness 优化，76.4% 超过所有手工调优。

关键是底下这句——这些改动每一个都超过同期"更好的模型"在同 benchmark 上 2-4 个百分点的典型涨幅。也就是说：变量是 harness，不是模型。

讲完这一页，听众会觉得"这件事不是噱头"——是有学术分量的、有数据的。

// 2. etclovg taxonomy

ETCLOVG · 7 层分类法

Li et al. 2026 (TMLR) 综合 170+ 开源项目提出的统一分类。把"Claude Code 5 层架构"提到了通用学术框架层级。

Observability 和 Governance 升级为独立层——以前被埋在 lifecycle hook 里。现实中各有独立工具栈，由不同团队维护。

同一篇 survey 的另一个核心贡献——ETCLOVG 七层分类法。

它把 agent harness 拆成 7 层，前面 4 层（E T C L）是结构主干，后面 3 层（O V G）是控制平面。

E · Execution & Sandbox：Agent 代码在哪跑，沙盒约束是什么。容器、computer-use、OS 权限。

T · Tool Interface：协议（MCP、A2A）、工具描述、工具发现、工具训练。

C · Context & Memory：短期窗口 / 跨 session 状态 / 长期 memory。100+ turns 的连贯性。

L · Lifecycle & Orchestration：单 agent loop / 多 agent / issue → PR pipeline。开源项目最多的一层（47 个）。

O · Observability：trace、成本、失败信号。Langfuse、OpenTelemetry。

V · Verification & Evaluation：benchmark、invariant test、trace-native eval。

G · Governance & Security：权限、身份、审计、prompt injection 防护。

不用记每一层。记 3 件事：一是"4 主干 + 3 控制平面"的结构；二是 L 层（生命周期）开源项目最多说明大家都在卷这块；三是 O 和 G 在开源里覆盖最稀缺，这就是机会——师弟师妹想做研究方向的可以看这两块。

// 3. maturity

用好 AI 工具的 3 个成熟度层级

层级典型行为典型 prompt瓶颈
L1 · 打字机 每次描述任务，看一行结果，改一行 "帮我写一个 X" 每次都从零开始
L2 · 协作者 用 skill / spec / 模板，让 AI 按既定流程做事 "按 brainstorming 流程帮我把 idea 变成 spec" 仍需要人介入每一步
L3 · 系统设计师 写 CLAUDE.md / 模板 / hook，Agent 自主完成多步任务 "按仓库规则消化今天的 HF papers" 需要前期投资搭 harness

大多数人卡在 L1。
今天目标：从 L1 跳到 L2，并理解 L3 怎么搭。

这页是今天最有诊断意义的一页。"用好 AI" 不是布尔值，是个层级。

L1 · 打字机。最常见的状态——每次想到任务就打 prompt，看一行结果，改一行。典型 prompt 是 "帮我写一个 X"。瓶颈是每次都从零开始，背景要重讲，偏好要重说。我估计在座大多数同学还在这个层级。

L2 · 协作者。开始用 skill、用 spec、用模板，让 AI 按既定流程做事。典型 prompt 是 "按 brainstorming 流程帮我把 idea 变成 spec"——你不再描述 "做什么"，而是引用 "用什么流程做"。瓶颈是仍需要人介入每一步，但是质量已经稳定多了。

L3 · 系统设计师。写 CLAUDE.md、模板、hook，让 Agent 在你预设好的环境里自主完成多步任务。典型 prompt 是 "按仓库规则消化今天的 HF papers"——这一句话背后是 6 步流程已经写在 CLAUDE.md 里。瓶颈是前期要投资搭 harness，但搭好之后回报指数级。

大多数师弟师妹卡在 L1。今天的目标，是让你们出门后能跳到 L2，并理解 L3 怎么搭。

// 3. prompt templates

三组立刻可以抄的 prompt 模板

templ-1 · 读论文（替代 "总结这篇 paper"） # 读论文 spec-driven prompt 请按以下结构总结 [arXiv:XXXX.XXXXX] 这篇论文： 1. 一句话定位：它在做什么、与最相关的 1-2 项前序工作的差异 2. 关键 insight：1-3 条，每条 ≤ 30 字 3. 方法核心：用 1 张架构图说明 + 不超过 200 字 4. 实验 takeaway：3 个最关键数字（带引用页码） 5. 局限：作者自己承认的 + 我应该追问的 6. 如果我要复现，最小可行版本是什么 # 按仓库命名规则保存
templ-2 · 写笔记（替代 "帮我总结一下 X"） # 不要直接动手，先 brainstorm 笔记的"用途" 我想就 [topic / paper / 实验] 写一份笔记。先不要动手。 先一问一答帮我把以下 5 件事确定下来： 1. 这份笔记将来谁会读？（自己 3 个月后？合作者？导师？） 2. 笔记的用途：备忘 / 引用源 / 综述素材 / 复现 SOP？ 3. 我想沉淀的核心结论（不是知识点，是结论） 4. 笔记结构草稿（标题 + 一级小标题） 5. 需要外部引用 / 数据 / 图的位置（先列 placeholder） 确定后再 /writing-plans，再写。
templ-3 · 跑实验（替代 "帮我实现 X"） # TDD-first，强制自我验证 我要实现 [algorithm]。请按 test-driven-development skill： 1. 先列 5 条数学/物理 invariant（必须满足的性质） 2. 先写 pytest 测试覆盖这 5 条 invariant 3. 再写实现 4. 跑测试 → 把完整输出贴回来（不是"测试通过"四个字） 5. 跑完后启动一个 sub-agent reviewer， 让它批判性地审查，找数学错误

这一页是今天最具体、最能立刻动手的一页。三组 prompt 模板，各自替代一类常见的烂 prompt。

第一组——读论文。把 "总结这篇 paper" 这种空 prompt，换成结构化的 6 段提示。重点在第 6 条——"如果要复现，最小可行版本是什么"。这一条逼 AI 不停留在 abstract 层，而是真正啃方法。

第二组——写笔记。把 "帮我总结一下" 换成 "先不要动手，先一问一答确定 5 件事"。这就是 spec-driven。第 1 条和第 2 条最关键——谁读 + 用途。给自己 3 个月后看的备忘 和 给导师看的汇报 写法完全不同。AI 不知道这件事，你必须告诉它。第 3 条核心结论——不要让笔记沦为知识点堆砌，要有"我从这件事得出 X"的句子。

第三组——跑实验。把 "帮我实现 X" 换成 5 步 TDD。重点在第 1 条——"先列 5 条数学 invariant"。让 AI 先把 "什么叫做对" 写成可测试的代码，再去实现。第 5 条是 sub-agent reviewer——别让 AI 自己评。

这三组模板我会发给大家。回去最快的练习——明天读你的下一篇 paper 时用第一组。

// stage ⑤ · evaluate

⑤ Evaluate · 三层评测防自评偏差

核心问题：模型表面 work 不等于真 work。

L1
标准 benchmark

lm-evaluation-harness（mmlu/hellaswag/...）· bigcode-evaluation-harness（HumanEval）· nemo-evaluator · 跨模型可比

L2
Spec invariant tests

从 ③ spec 里的 N 个 invariant 派生 pytest，覆盖数学/物理性质。"测试通过 4 个字" 不算证据，要贴完整输出。

L3
Sub-agent reviewer

独立 sub-agent 看输出样本，找定性问题。"Out of the box, Claude is a poor QA agent"——必须用独立 Agent 评。

三层都过 = 真 work。
只过 L1 = 表面 work（90% 的"work"其实在这一档）。

第五阶段 Evaluate，问题是——表面 work 不等于真 work。

这是整个 pipeline 最容易出"假阳性"的一段。我提的方案是三层评测，三层都过才算 work。

L1 标准 benchmark——lm-evaluation-harness 跑 mmlu / hellaswag / gsm8k 这些跨模型可比的指标。这一层最容易跑，也最不能停在这里。

L2 spec invariant tests——把第三阶段 spec 里写的 N 个 invariant 落成 pytest，测数学和物理性质。比如做 RoPE 实验就测旋转不变性、测周期性。这一层会暴露 90% 的"代码有 bug 但 benchmark 通过"。

L3 sub-agent reviewer——起一个独立 agent 看模型输出样本，找定性问题。这一层是 Anthropic Generator-Evaluator 模式的落地。

底下这句话很狠——三层都过 = 真 work，只过 L1 = 表面 work。大家觉得别人发的 paper 里 90% 的"work"其实在 L1 这一档。这就是为什么很多 paper 复现不出来。

// 4. the unifier

把 6 阶段串起来：
Artifact Registry = pipeline 总线

Artifact Registry = 任何能给产物提供"稳定 ID + 元数据 + 跨阶段引用"的系统。

把刚才 6 阶段串起来看，会发现一件之前没强调的事——所有阶段都需要一种"产物注册中心"来当总线。

我把这种系统抽象叫 Artifact Registry。它的定义是——任何能给产物提供"稳定 ID + 元数据 + 跨阶段引用"的系统。

它给 pipeline 提供三件关键基础设施。

第一，稳定 ID。重点是稳定——不是文件路径，因为文件路径会搬家、会失效；是跨机器、跨时间不变的引用。任何阶段都可以靠 ID 互相引用。

第二，机器可读 provenance。每个产物有 base_model、dataset、commit、config 字段，能反查"这个产物从哪来"。这是"可审计"的物理基础。

第三，跨阶段闭环——你 ① 阶段读到的 paper，往往链接到 ② 的 dataset、③ 的 model、④ 的 demo。研究的输入和输出是同一类对象。

下半页是实例对照表。这一页的关键在于——不要绑死在某一个具体平台上。

NLP 大模型场景，HF Hub 是事实标准；CV 或通用 ML，GitHub Releases 加自有 S3 也行；实验中间产物推荐 W&B Artifacts 或 MLflow；国内合规需求或中文社区可以用 ModelScope；课题组内部最简单的方案是 DVC 加一个 manifest.json。

底下这句话是这一页的精神——关键不是用哪个 Registry，而是"凡是产物必须有稳定 ID"这个纪律。师弟师妹的课题组用什么、有什么合规要求、跨平台不跨平台——选 Registry 时考虑这些。但"产物有稳定 ID"这件事不能省。

// 4. before vs after

实验流程对比 · 从碎片到管线

这一页是今天最有冲击力的一页。我把"以前怎么做实验"和"现在怎么做实验"并排放出来。

看左边——这是绝大多数硕士生的真实状态。"我觉得 X 可能 work"，立刻动手写代码，超参藏在 colab cell 里，跑通就发微信给导师，跑一个 benchmark 看起来比 baseline 高就算赢，写进周报就完事。三个月后自己都看不懂自己做了什么——这就是"散点式实验"的代价。

看右边——同样的实验，但走 spec-driven 管线。从 DDD（Domain-Driven Design）开始：你为什么做 X？前序工作是什么？假设是什么？这一步逼你回答的问题，AIDLC 把它叫"What should we build?"。然后到 SDD（Spec-Driven Development）——spec.yaml 是 single source of truth，hypothesis、metric、invariant、各产物的 registry ID 全在这一份 yaml 里。再到 TDD——先写 invariant 测试再写实现，AIDLC 的话是"tests before code, not after"。然后三层评测、四件套发布、最后 reflect 把失败也写回知识库。

这套 DDD + SDD + TDD 不是我自己编的——是 AWS AIDLC 团队在 Amazon 内部 Frontier Teams 验证过的方法论栈，他们在工程项目上做到了 6 倍生产力。我把它搬到科研场景。

底下两句对比要慢——左边"上下文藏在脑子里 · 判据靠'我觉得对'"，右边"spec 是契约 · 测试是证据 · 反思可累积"。这就是从"个人手工业"到"科学管线"的差距。

// 4. why is this scientific

为什么这个流程才叫科学？

不是因为用了 AI——而是因为它满足科研方法论的5 个基本属性，每条都由具体 harness 部件保证：

反向自查：① 别人能复现吗？ ② 失败/成功有明确判据吗？ ③ 决策链有文字记录吗？ ④ 教训沉淀到团队了吗？ ⑤ 能跟同方向横向比吗？

上一页讲了"以前 vs 现在"的流程对比。这一页回答更深的问题——为什么右边那个流程才叫"科学"？

科学方法论本身有 5 个基本属性——可重复、可证伪、可审计、可累积、可比较。这些不是 AI 时代发明的，是从 17 世纪科学革命就定下来的标准。

但师弟师妹们做实验时，这 5 条往往只靠"自觉"在维持——你尽量记笔记、尽量多跑几个 seed、尽量写清楚——但因为是手工，常常只做到一半就放弃了。

新流程的关键不是"快"，是把这 5 条从自觉变成机械化保证。我一条一条说——

① 可重复：spec.yaml 把 base model、dataset、超参全部锁定到稳定的 registry ID（HF / GitHub / W&B / 内部都行）。别人 git clone 你的 spec，就能复现你的实验。这件事手工做永远做不完整，自动化一次到位。

② 可证伪：TDD 的 invariant 测试把"什么叫做对"写成可执行代码。RoPE 实验，你必须先写"旋转不变性测试"——这就是 Karl Popper 说的"可证伪边界"用代码定义出来。

③ 可审计：DDD 文档记录每个决策为什么这么做，wandb 记录每个 run，sub-agent reviewer 留 log——整条 audit trail 完整。"为什么做这个实验"、"为什么改了这个超参"——以前都是脑子里的，现在都在文件里。

④ 可累积：REFLECT 阶段把失败原因写回 DDD 文档。下次做相关实验时，AI 会自动读这份 DDD，先帮你避坑——这是 AIDLC 提到的"self-evolving knowledge layer"。经验不再只在脑子里、不再只在毕业的师兄师姐脑子里。

⑤ 可比较：三层评测加标准 metric，让你的实验和同方向其他工作能横向比。

底下这句话是这一页的精神——"AI 加速科研"的真正意义不是跑得快，是让 5 个属性从自觉变成机械化保证。

最后留一个反向自查给师弟师妹——你最近一次实验，能勾几个？ 这一段如果能给大家留下"我得回去检查一下"的不安感，今天就成了。

周末就能搭起来的 4 件套，不需要写任何代码。

第一件，CLAUDE.md。15 分钟一份初版。一份成熟的 CLAUDE.md 通常包括项目说明、命名规则、关键工作流、外部工具配置——一两百行就够了。Agent 一进入项目就先读它。

第二件，MEMORY.md 加 memory/ 目录。这是从 L1 跳到 L2 最关键的一步——你不再靠每次重复说话来传递背景。一个典型的 MEMORY.md 只是个几行索引，指向几个具体记忆文件——身份、写作风格、常用工具配置、引用偏好。

第三件，模板文件。论文笔记一个、研究笔记一个、周报一个、综述一个。Anthropic V1 时代就强调：用 JSON 而非 Markdown 存任务清单，模型不容易乱改——背后逻辑就是结构性强，AI 才能做得一致。

第四件，一个 shell 脚本，把"建笔记、刷索引、搜索"这些常用动作固化成命令。Agent 调一次脚本比让它每次重写 50 行 bash 稳定多了。

用 Anthropic Justin Young 这句话收尾——找最简单可行的方案，只在需要时再加复杂度。不要一上来就抄 OpenAI 的百万行架构，从这 4 件套开始就够了。

// 6. concrete example

CLAUDE.md 长什么样（通用研究项目模板）

# <your-research-project> · CLAUDE.md ## What This Is A research project on [topic]. Markdown notes organized into folders, managed by a shell CLI. ## Naming Conventions | Folder | Pattern | Example | |---|---|---| | papers/ | YYYY-short-title.md | 2026-rope-encoding.md | | notes/ | YYYY-MM-DD-title.md | 2026-MM-DD-experiment-X.md | ## Paper Reading Workflow 1. Fetch metadata: Get arXiv ID, abstract, authors 2. Deduplicate: Skip if already in papers/ 3. Read: Apply the 6-section reading template (see templ-1) 4. Download figures: Save to papers/<slug>/ subfolder 5. Write summary: Use _template.md as starting point 6. Update indexes: Run bash tools.sh index

这就是 Spec-as-Code——把工作流写成 markdown，Agent 直接当代码执行。

给大家看一份通用模板的 CLAUDE.md 节选。

顶上 "What This Is" 一句话告诉 Agent 这是什么项目。

中间 "Naming Conventions" 一张表告诉它命名规则——这就是上一幕讲的"机械化约束"。

底下 "Paper Reading Workflow" 6 步流程，每步都是动词开头，每步都有具体动作（API 调用、子目录组织、模板使用、命令执行）。Agent 看到这个就直接执行。

这个文件本质上是 Spec-as-Code——把工作流写成 markdown，但它不是文档，是给 Agent 的可执行 SOP。
